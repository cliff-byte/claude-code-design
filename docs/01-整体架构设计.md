# Claude Code 整体架构设计

> 从 1884 个源文件中提炼的系统全貌，分析一个顶级 AI Agent 产品是如何组织的。

## 一、系统分层总览

Claude Code 采用**五层解耦架构**，从上到下依次为：

```mermaid
graph TB
    subgraph L0["Layer 0: 入口与启动"]
        CLI["cli.tsx<br/>动态路由 · 快速路径"]
        INIT["init.ts<br/>配置 · 认证 · 网络"]
        BOOT["bootstrap/state.ts<br/>全局单例状态"]
    end

    subgraph L1["Layer 1: UI 与交互"]
        INK["ink/<br/>终端 UI 框架"]
        COMP["components/<br/>React 组件库"]
        HOOKS["hooks/<br/>100+ React Hook"]
        STATE["state/<br/>Zustand Store"]
    end

    subgraph L2["Layer 2: 查询与协调"]
        QE["QueryEngine<br/>AI 查询主循环"]
        QUERY["query.ts<br/>对话循环 · 工具编排"]
        COORD["coordinator/<br/>多 Worker 协调"]
        CMD["commands/<br/>100+ 斜杠命令"]
    end

    subgraph L3["Layer 3: 工具层"]
        TOOLS["tools/<br/>40+ 工具实现"]
        TOOLEXEC["toolExecution.ts<br/>权限 · 进度 · Hook"]
    end

        API["api/<br/>Claude API 封装"]

    end

    subgraph L5["Layer 5: 基础设施"]
        UTILS["utils/<br/>331+ 工具函数"]

    end

    L0 --> L1
    L1 --> L2
    L2 --> L3
    L3 --> L4
    L4 --> L5
```

### 每一层在做什么？

| 层级 | 核心职责 | 定位 |
|------|---------|------|
| **Layer 0** 入口与启动 | 解析命令行参数，初始化配置、认证、网络 | 系统引导与环境准备 |
| **Layer 1** UI 与交互 | 终端界面渲染、用户输入处理、状态管理 | 用户交互界面层 |
| **Layer 2** 查询与协调 | 驱动 AI 对话循环、分发工具调用、协调多个 Agent | 核心决策与调度中枢 |
| **Layer 3** 工具层 | 40+ 种具体能力的实现（读写文件、执行命令、搜索等） | AI 的执行能力层 |
| **Layer 4** 服务层 | API 调用、MCP 集成、上下文压缩、记忆管理 | 外部对接与数据管理 |
| **Layer 5** 基础设施 | 权限检查、文件操作、任务框架等底层能力 | 底层支撑设施 |

## 二、启动流程

Claude Code 的启动经过精心优化，让简单命令（如 `--version`）毫秒级响应，复杂功能按需加载。

```mermaid
flowchart TD
    START["用户执行 claude 命令"] --> FAST{"快速路径？"}
    
    FAST -->|"--version"| VER["直接输出版本<br/>零模块加载"]
    FAST -->|"--help"| HELP["输出帮助信息"]
    FAST -->|"其他"| MAIN["进入 main.tsx"]
    
    MAIN --> PARALLEL1["并行初始化"]
    
    PARALLEL1 --> C1["加载配置<br/>settings.json"]
    PARALLEL1 --> C2["检测 Git 仓库"]
    PARALLEL1 --> C3["预连接 API"]
    PARALLEL1 --> C4["检测 IDE 环境"]
    PARALLEL1 --> C5["加载远程设置"]
    
    C1 & C2 & C3 & C4 & C5 --> REACT["初始化 React 应用"]
    
    REACT --> APP["AppStateProvider<br/>↓<br/>MailboxProvider<br/>↓<br/>主应用组件"]
    
    APP --> WAIT["等待用户输入"]
    
    style FAST fill:#f9f,stroke:#333
    style PARALLEL1 fill:#bbf,stroke:#333
```

### 关键设计决策

| 决策 | 做法 | 效果 |
|------|------|------|
| **快速路径** | `--version` 在模块加载前就返回 | 毫秒级响应 |
| **并行初始化** | 配置/Git/API/IDE 检测同时进行 | 启动时间减半 |
| **延迟加载** | 命令和工具按需 import | 减少初始内存占用 |
| **特性门控** | 编译时移除未启用特性的代码 | 发布包体积更小 |

## 四、两层状态管理

Claude Code 采用了"两层状态"设计，把不同生命周期的数据分开管理：

```mermaid
graph TB
    subgraph GLOBAL["全局状态 (Bootstrap State)"]
        direction LR
        G1["totalCostUSD"]
        G2["cwd"]
        G3["sessionId"]
        G4["apiKey"]
    end

    subgraph SESSION["会话状态 (AppState / Zustand)"]
        direction LR
        S1["mainLoopModel"]
        S2["tasks"]
        S3["mcp.clients"]
        S4["toolPermissionContext"]
    end

    GLOBAL -->|"跨会话持久"| 磁盘
    SESSION -->|"会话内响应式"| UI

    style GLOBAL fill:#ffe0b2
    style SESSION fill:#b3e5fc
```

| 维度 | 全局状态 (Bootstrap) | 会话状态 (AppState) |
|------|---------------------|-------------------|
| **生命周期** | 进程级，跨会话 | 单次会话 |
| **更新方式** | 命令式 `setCwd(...)` | 响应式 `useAppState(s => ...)` |
| **用途** | 元数据、成本、CWD | UI 驱动、任务、权限 |
| **约 100 个变量** | 简单键值对 | 结构化状态树 |

## 六、核心执行路径

一次用户交互从输入到输出，经过以下路径：

```mermaid
sequenceDiagram
    participant U as 用户
    participant CLI as CLI 入口
    participant QE as QueryEngine
    participant API as Claude API
    participant TE as 工具执行器
    participant T as 具体工具

    U->>CLI: 输入消息
    CLI->>QE: submitMessage()
    
    loop 对话循环
        QE->>API: 发送消息 (流式)
        API-->>QE: 流式返回文本/工具调用
        
        alt 有工具调用
            QE->>TE: 执行工具
            TE->>TE: 检查权限
            TE->>T: 调用 tool.call()
            T-->>TE: 返回结果
            TE-->>QE: 工具结果
            Note over QE: 继续循环
        else 无工具调用
            QE-->>CLI: 返回最终响应
        end
    end
    
    CLI-->>U: 显示结果
```

### 路径说明

1. **输入路径**: CLI → processUserInput → QueryEngine → query()
2. **API 路径**: query() → buildMessages → Claude API (流式) → 解析响应
3. **工具路径**: 提取 tool_use → 权限检查 → Hook → 执行 → 后处理
4. **输出路径**: 工具结果 → 作为新消息追加 → 继续循环或退出

## 七、特性门控与条件编译

Claude Code 支持内部（Ant）和外部两种构建，通过特性门控在编译时移除不需要的代码：

```typescript
// 编译时特性检查 — 未启用的代码会被"死代码消除"
if (feature('COORDINATOR_MODE')) {
  // 仅内部构建包含此代码
  const coordinatorMode = require('./coordinator/coordinatorMode.js')
}
```

| 特性 | 内部构建 | 外部构建 | 说明 |
|------|---------|---------|------|
| COORDINATOR_MODE | ✅ | ❌ | 多 Worker 协调模式 |
| REPL_TOOL | ✅ | ❌ | 代码 REPL |
| CACHED_MICROCOMPACT | ✅ | ❌ | 缓存微压缩 |
| MCP | ✅ | ✅ | MCP 协议 |
| PLUGINS | ✅ | ✅ | 插件系统 |

## 八、架构设计原则总结

| 原则 | 体现 |
|------|------|
| **渐进式初始化** | 快速路径→并行初始化→按需加载 |
| **最小全局状态** | Bootstrap ~100 变量，其余在 AppState |
| **工具是一等公民** | 统一接口，40+ 工具全部走相同管道 |
| **消息不可变** | 历史可重播，压缩可验证 |
| **流式优于批量** | 所有 API 调用默认流式，实时 UI 反馈 |
| **Hook 优于硬编码** | 所有关键节点都有可扩展的 Hook |
| **特性门控优于条件分支** | 编译时移除，零运行时开销 |
| **协调不递归** | Worker 工具受限，防止无限嵌套 |
