# Claude Code 对话与数据流

> 一条消息从用户键盘到 AI 响应的完整流转路径分析。

## 一、消息类型全景

Claude Code 中所有交互都是"消息"。理解消息类型，就理解了整个数据流。

```mermaid
graph TB
    subgraph 用户侧消息
        UM["UserMessage<br/>用户输入的文本/图片"]
        TR["ToolResult<br/>工具执行结果（自动生成）"]
        ATT["AttachmentMessage<br/>附件/记忆文件"]
    end

    subgraph AI 侧消息
        AM["AssistantMessage<br/>AI 的回复"]
        TU["ToolUse<br/>AI 请求调用工具"]
        TH["Thinking<br/>AI 的思考过程"]
    end

    subgraph 系统消息
        SM["SystemMessage<br/>内部流程控制"]
        CB["CompactBoundary<br/>压缩分界标记"]
        PM["PermissionRetry<br/>权限重试"]
    end

    UM --> AM
    AM --> TU
    TU --> TR
    TR --> AM
    AM --> TH

    style UM fill:#e8f5e9
    style AM fill:#e3f2fd
    style SM fill:#fff3e0
```

### 消息类型详解

| 消息类型 | 角色 | 包含内容 | 发送给 API？ |
|---------|------|---------|------------|
| **UserMessage** | user | 文本、图片、粘贴引用 | ✅ |
| **AssistantMessage** | assistant | 文本、工具调用、思考过程 | ✅ |
| **ToolResult** | user | 工具执行结果（stdout、文件内容等） | ✅ |
| **AttachmentMessage** | system | 记忆文件、上下文附件 | ✅ |
| **CompactBoundary** | system | 压缩分界线（标记压缩事件） | ❌ 仅内部 |
| **PermissionRetry** | system | 权限被拒后的重试标记 | ❌ 仅内部 |

### 场景：用户输入 "列出 src 目录的文件"

```mermaid

    participant 输入处理
    participant 查询循环
    participant Claude_API
    participant 工具执行器
    participant Bash工具

    用户->>输入处理: "列出 src 目录的文件"
    
    Note over 输入处理: 展开粘贴引用<br/>加载附件<br/>生成 UserMessage

    输入处理->>查询循环: submitMessage()
    
    rect rgb(230, 240, 255)
        Note over 查询循环: 迭代 1: AI 决定调用工具
        查询循环->>查询循环: 检查是否需要压缩
        查询循环->>查询循环: 构建系统提示 + 消息数组
        查询循环->>Claude_API: 流式请求
        Claude_API-->>查询循环: 流式返回: tool_use(Bash, "ls src")
        查询循环->>工具执行器: 执行 Bash("ls src")
        工具执行器->>工具执行器: 权限检查 → 允许
        工具执行器->>Bash工具: call("ls src")
        Bash工具-->>工具执行器: "main.tsx\nquery.ts\ntools.ts..."
        工具执行器-->>查询循环: ToolResult
        Note over 查询循环: 追加工具结果到消息数组<br/>继续循环
    end

    rect rgb(230, 255, 230)
        Note over 查询循环: 迭代 2: AI 生成最终回复
        查询循环->>Claude_API: 流式请求（含工具结果）
        Claude_API-->>查询循环: 流式返回: 文本回复
        Note over 查询循环: 无工具调用 → 退出循环
    end

    查询循环-->>用户: "src 目录包含以下文件:..."
```

## 三、查询循环状态机

`query.ts` 是整个系统的心脏，它是一个**状态机驱动的循环**：

```mermaid
stateDiagram-v2
    [*] --> 构建消息
    构建消息 --> 检查压缩: 每次迭代开始
    检查压缩 --> 调用API: 未超阈值
    检查压缩 --> 执行压缩: 超过阈值
    执行压缩 --> 调用API: 压缩完成
    
    调用API --> 解析响应: 流式接收
    
    解析响应 --> 执行工具: 有 tool_use
    解析响应 --> 退出循环: 无 tool_use（正常结束）
    解析响应 --> 恢复重试: max_output_tokens 错误
    
    执行工具 --> 构建消息: 工具结果 → 继续循环
    恢复重试 --> 调用API: 调整参数重试（最多3次）
    
    退出循环 --> [*]
```

### 循环继续的5种原因

| 继续类型 | 触发条件 | 行为 |
|---------|---------|------|
| `tool_result_continue` | AI 调用了工具 | 执行工具，把结果追加到消息，继续 |
| `max_output_tokens_recovery` | 输出超长被截断 | 减少 max_tokens，重试（最多3次） |
| `compact_recovery` | 主动压缩后继续 | 压缩历史，用摘要替代，继续 |
| `reactive_compact_recovery` | prompt_too_long 错误 | 紧急压缩后重试 |
| `max_output_tokens_continue_on_error` | 错误恢复 | 尝试继续 |

## 四、API 交互详解

### 4.1 请求构建

```mermaid
graph LR
    MSG["消息数组"] --> NORM["normalizeMessagesForAPI<br/>格式转换 · 图片验证"]
    SYS["系统提示"] --> SPLIT["分割为静态/动态<br/>优化缓存"]
    TOOLS["工具定义"] --> SCHEMA["toolToAPISchema<br/>转 JSON Schema"]
    
    NORM & SPLIT & SCHEMA --> PARAMS["API 请求参数"]
    PARAMS --> API["POST /v1/messages<br/>流式调用"]
```

**请求参数构成**：

```
{
  model: "claude-sonnet-4-6",
  max_tokens: 32000,
  system: [
    { text: "静态指令...", cache_control: { type: "ephemeral" } },  ← 可缓存
    { text: "动态上下文..." }                                       ← 每次变化
  ],
  messages: [...],              // 转换后的消息
  tools: [...],                 // 工具 schema
  tool_choice: "auto",
  temperature: 1,
  stream: true
}
```

### 4.2 流式响应处理

```mermaid
sequenceDiagram
    participant API as Claude API
    participant 解析器
    participant UI as 终端 UI

    API->>解析器: message_start (初始化)
    API->>解析器: content_block_start (文本块开始)
    
    loop 文本流式
        API->>解析器: content_block_delta (文本片段)
        解析器->>UI: 实时渲染文本
    end
    
    API->>解析器: content_block_stop (文本块结束)
    
    opt 工具调用
        API->>解析器: content_block_start (tool_use 开始)
        loop 工具输入流式
            API->>解析器: input_json_delta (JSON 片段)
        end
        API->>解析器: content_block_stop (tool_use 结束)
        解析器->>解析器: JSON.parse(累积的输入)
    end

    API->>解析器: message_stop (结束)
```

### 4.3 重试与回退

```mermaid

    OK -->|否| ERR{"错误类型？"}
```

## 五、上下文构建策略

系统提示由**三层内容**拼接而成：

```mermaid
graph TB
    subgraph 静态层["静态层 (可缓存 · 跨用户共享)"]
        S1["角色定义<br/>'你是 Claude Code...'"]
        S2["工具使用指导"]
        S3["编码风格要求"]
        S4["输出格式规范"]
    end

    subgraph BOUNDARY["═══ 缓存分界线 ═══"]
    end

    subgraph 动态层["动态层 (每次变化)"]
        D1["Git 状态<br/>分支 · 未提交更改"]
        D2["CLAUDE.md 指令"]
        D3["当前日期"]
        D4["MCP 服务器指令"]
        D5["记忆文件"]
        D6["环境信息<br/>OS · Shell · 模型"]
    end

    静态层 --> BOUNDARY --> 动态层
```

### 为什么要分两层？

**提示缓存**是 Claude API 的重要优化特性：

- **静态层**加了 `cache_control` 标记 → 跨请求复用，节省 token 费用
- **动态层**每次都不同 → 不缓存，但也不会影响静态层的缓存命中

这个设计让 Claude Code 的 API 调用成本**大幅降低**。

## 六、Compact 压缩机制

对话越来越长，上下文窗口终会不够用。Compact 就是"智能遗忘"机制。

### 6.1 三层压缩策略

```mermaid
graph TB
    subgraph 第一层["微压缩 (MicroCompact)"]
        MC1["清理旧工具结果"]
        MC2["超过24h的结果 → '[Old content cleared]'"]
        MC3["频率: 每次 API 调用前"]
    end

    subgraph 第二层["自动压缩 (AutoCompact)"]
        AC1["用 AI 生成对话摘要"]
        AC2["保留最近消息"]
        AC3["触发: token 超过阈值"]
    end

    subgraph 第三层["反应式压缩 (Reactive)"]
        RC1["紧急压缩"]
        RC2["触发: API 返回 prompt_too_long"]
    end

    第一层 -->|"仍然超"| 第二层
    第二层 -->|"仍然超"| 第三层
```

### 6.2 何时触发压缩？

```
模型上下文窗口（如 200K tokens）
│
├─── 有效窗口 = 窗口 - max_output_tokens
│    （200K - 32K = 168K）
│
├─── 自动压缩线 = 有效窗口 - 13K 缓冲
│    （168K - 13K = 155K）
│    → 超过这条线：触发完整压缩
│
├─── 警告线 = 自动压缩线 - 20K
│    （155K - 20K = 135K）
│    → 超过这条线：显示警告
│
└─── 阻塞线 = 窗口 - 3K
     （200K - 3K = 197K）
     → 超过这条线：必须先压缩才能继续
```

### 6.3 压缩后恢复

压缩不仅是"删除旧消息"，还会**恢复关键上下文**：

```
压缩完成后：
├─ 恢复最近 5 个编辑过的文件（每个最多 5K tokens）
├─ 重新注入活跃的 Skill 指令（最多 25K tokens）
├─ 附加 MCP 指令变化
└─ 总恢复预算: 50K tokens
```

## 八、Token 使用与成本追踪

每次 API 调用都会记录 token 使用：

```
usage: {
  inputTokens: 1500,            // 输入 token
  outputTokens: 200,             // 输出 token
  cacheReadInputTokens: 12000,   // 缓存命中 (大幅便宜)
  cacheCreationInputTokens: 500, // 缓存写入
  costUSD: 0.0043                // 美元成本
}
```

### Token 估算策略

| 方法 | 精度 | 速度 | 用途 |
|------|------|------|------|
| **API 精确计数** | 100% | 慢（需要 API 调用） | 关键决策（是否压缩） |
| **Haiku 回退** | ~95% | 中 | API 不可用时 |
| **粗略估算** | ~80% | 极快 | 实时 UI 显示 |

粗略估算规则：
- 普通文本：**4 字节 ≈ 1 token**
- JSON 数据：**2 字节 ≈ 1 token**（JSON 更密集）
- 图片：固定 **2000 tokens**

## 九、错误处理全景

```mermaid
flowchart TD
    ERR["API 错误"] --> TYPE{"错误类型"}
    
    TYPE -->|"max_output_tokens"| MOT["减少 max_tokens<br/>重试最多 3 次"]
    TYPE -->|"prompt_too_long"| PTL["触发反应式压缩<br/>压缩后重试"]
    TYPE -->|"rate_limit (429)"| RL["检查配额<br/>等待后重试"]
    TYPE -->|"overloaded (529)"| OL["指数退避<br/>最多 2 次重试"]
    TYPE -->|"authentication"| AUTH["提示重新登录"]
    TYPE -->|"billing"| BILL["提示检查账单"]
    TYPE -->|"server_error"| SE["重试<br/>→ 非流式<br/>→ Haiku 回退"]
    
    MOT -->|"3次仍失败"| FAIL["放弃，报告错误"]
    PTL --> CONTINUE["压缩后继续对话"]
```

## 十、设计亮点总结

| 设计点 | 做法 | 为什么 |
|--------|------|--------|
| **流式优先** | 所有 API 调用默认流式 | 用户看到实时输出，体验好 |
| **状态机循环** | query.ts 用 while(true) + transition | 支持工具调用、错误恢复、压缩等多种继续方式 |
| **提示缓存二分** | 静态/动态分界线 | 最大化缓存命中率，降低成本 |
| **三级压缩** | 微压缩 → 自动压缩 → 反应式 | 渐进式应对，避免过度压缩 |
| **压缩后恢复** | 恢复最近文件和 Skill | 压缩不丢关键上下文 |
| **多级回退** | 流式 → 非流式 → Haiku | 最大化可用性 |
| **JSONL 持久化** | 行追加，无需读-改-写 | 原子写入，不怕崩溃 |
